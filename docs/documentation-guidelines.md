# 文档维护规范

这份文档面向仓库维护者，只回答一个问题：

`super-stack` 里不同类型的信息，应该写到哪里，怎么保持不打架。

目标不是增加文档数量，而是减少这些常见问题：

- README 越写越长，最后变成设计稿
- `docs/` 里重复解释已经在首页说过的内容
- `.planning/` 里既写当前状态，又混入长期设计说明
- 文档更新了，但脚本路径、测试入口、CI 口径没有一起同步

## 1. 文档分层原则

默认按这 4 层写：

1. `README.md`
   - 回答“是什么、怎么装、怎么验、去哪里看细节”
2. `docs/*.md`
   - 回答“为什么这样设计、具体规则是什么、长期边界是什么”
3. `.planning/*.md`
   - 回答“当前项目状态是什么、下一步做什么、当前风险是什么”
4. `.planning/codebase/*.md`
   - 回答“当前仓库事实长什么样、结构/测试/架构/集成现状如何”

## 2. README 写什么

`README.md` 只保留高频入口信息：

- 项目定义
- 产品边界
- 安装命令
- 检查与回归命令
- 测试入口
- 文档导航

`README.md` 不应该承载：

- 大段设计推导
- 候选技术方案历史
- 长篇路线图
- 详细脚本分层解释
- 阶段性整改记录

判断标准很简单：

- 如果用户第一次打开仓库 3 分钟内必须知道，就放 README
- 如果不是第一次就必须看到的内容，就不要堆在 README

## 3. docs 写什么

`docs/` 用来放长期有效、可复用、可单独阅读的说明。

适合放进 `docs/` 的内容：

- 架构设计
- 脚本分层规则
- 测试分层策略
- 浏览器方案取舍
- hooks 规则与边界
- 参考项目调研
- 文档维护规则本身

`docs/` 里的文档应尽量满足：

- 单篇只回答一类问题
- 能脱离当前会话单独阅读
- 不依赖“Phase 1/2/3 刚完成”这类时态
- 尽量少写“这轮”“当前先”“之后再说”这类短期表述

## 4. .planning 写什么

`.planning/` 是项目内工作状态，不是公共说明书。

适合放：

- 当前 focus
- 当前风险
- 下一步动作
- 当前优先级
- 当前决策

不适合放：

- 面向仓库读者的长期设计说明
- 大量重复 README / docs 的内容
- 已经完成但不再影响当前判断的长篇历史过程

推荐分工：

- `ROADMAP.md`: 当前结构、当前优先级、后续方向
- `STATE.md`: 当前状态、当前风险、下一步动作
- `REQUIREMENTS.md`: 仍然有效的硬约束
- `PROJECT.md`: 最短项目定义

## 5. .planning/codebase 写什么

`.planning/codebase/` 记录的是“仓库事实地图”，不是规划。

适合放：

- 当前结构
- 当前测试布局
- 当前架构形态
- 当前集成点
- 当前约定
- 当前问题与风险

这里的内容要尽量：

- 贴近实际文件与目录
- 避免写成愿景
- 避免写成历史整改过程

## 6. 文档更新的同步规则

出现下面这些变化时，文档必须同步更新：

### 6.1 脚本入口变化

如果脚本路径、入口名称、目录分层变化，至少同步更新：

- `README.md`
- `docs/script-architecture.md`
- `docs/validation-strategy.md`
- `.planning/codebase/STRUCTURE.md`
- `.planning/codebase/TESTING.md`
- CI 配置里的调用路径

### 6.2 测试策略变化

如果 `unit / integration / smoke` 的职责或入口变化，至少同步更新：

- `README.md`
- `docs/validation-strategy.md`
- `.planning/codebase/TESTING.md`
- `.github/workflows/ci.yml`

### 6.3 产品边界变化

如果产品边界变化，例如是否支持项目级安装、是否引入新的默认浏览器主链路，至少同步更新：

- `README.md`
- `docs/project-design.md`
- `docs/browser-technology-options.md`
- `.planning/REQUIREMENTS.md`
- `.planning/ROADMAP.md`
- `.planning/STATE.md`

## 7. 文档写作风格规则

默认遵守这些规则：

- 面向用户可见文档优先中文
- 技术名词、命令、路径、协议名保留英文
- 一篇文档只服务一个主要目的
- 优先写“当前有效事实”，少写阶段性口号
- 避免把同一段解释复制到 3 份文档里

如果一个内容已经在别处完整存在，优先：

1. 在当前文档里给出一句结论
2. 链接到主文档

而不是重复粘贴整段说明

## 8. 文档是否该新增的判断标准

新增文档前先问 3 个问题：

1. 这个内容是否已经能放进现有文档的某一节？
2. 它是否真的代表一类独立问题，而不是一次性备注？
3. 它是否会在后续被持续维护？

只有当答案偏向“是”，才值得新建文档。

否则优先补现有文档。

## 9. 推荐维护流程

每次结构性改动后，按这个顺序检查：

1. 代码 / 脚本入口是否已经稳定
2. `README.md` 是否还只保留入口信息
3. 相关 `docs/*.md` 是否仍与实现一致
4. `.planning/ROADMAP.md` 和 `.planning/STATE.md` 是否仍反映当前状态
5. `.planning/codebase/*.md` 是否仍反映仓库事实

如果只愿意做一件最小同步动作，优先保证：

- README 不撒谎
- 文档导航能把人带到正确位置

## 10. 当前结论

`super-stack` 的文档体系应长期保持这个结构：

- `README`: 入口
- `docs`: 长期规则与设计
- `.planning`: 当前状态与决策
- `.planning/codebase`: 仓库事实地图

后续如果文档开始再次混写，优先不是继续补新文档，而是先把内容收回到正确层级。
