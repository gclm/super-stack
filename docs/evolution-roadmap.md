# super-stack 演进路线图

这份路线图不描述“理想状态下能做什么”，而描述“按当前架构和验证能力，下一步最值得怎么做”。

目标是：

1. 保持当前主线稳定
2. 避免功能堆积失控
3. 让每一阶段都能被验证

## 1. 路线图原则

后续演进遵循 4 条原则：

### 1.1 先补验证，再补能力

任何新增能力，如果没有最小验证路径，就不应该优先进入主线。

### 1.2 先补高频痛点，再补理论完整度

优先顺序应该是：

- 高频噪音
- 高频误判
- 高频交付缺口

而不是“看起来很完整”的大而全系统。

### 1.3 共享核心优先

如果一个能力只能在单宿主工作，先判断：

- 是否真的值得进入共享主线
- 是否应该先作为宿主局部扩展

### 1.4 文档必须与实现同步

后续每补一类能力，至少同步更新：

- 对应 `docs/`
- 安装说明
- 验证说明

## 2. 当前状态判断

按当前状态，`super-stack` 已经完成了“结构收敛 + 最小验证闭环”阶段，当前进入“稳定演进”阶段。

可以粗分为：

### 已完成

- 共享路由
- 核心 skills
- 全局安装主线
- Claude / Codex 双宿主接线
- 状态恢复 hooks
- readonly auto-allow hooks
- 安装检查
- 基础回归

### 当前重点

- 保持单一脚本入口结构不回退
- 继续补强 browser / Claude / Codex 的真实环境回归证据
- 维持 README、docs、ROADMAP、STATE 与实际实现同步
- 在不增加结构复杂度的前提下持续增强能力

## 3. 后续演进方向

后续不再是“大规模整改阶段”，而是围绕当前底座做持续增强。下面 4 个方向代表的是演进重点，而不是新的结构改造工程。

## Direction 1：稳定核心体验

目标：

- 让当前链路足够稳，不再频繁暴露“已经装了但不确定是否真的生效”的问题

### 重点任务

1. 清理剩余安装 / 检查误报
2. 继续补齐 README 与 docs 的一致性
3. 把当前回归脚本收成标准入口
4. 梳理 `docs/` 文档索引

### 完成标志

- 安装、检查、回归、文档四者对齐
- 关键能力都有明确入口

## Direction 2：稳定浏览器与 QA 链路

目标：

- 让 `browse` 不再只是“路由正确”，而是“真实可用于前端问题排查”

### 重点任务

1. 稳定 `super-stack-browser` 入口体验
2. 继续减少重复授权与会话漂移
3. 增加 browse 专项回归
4. 在 `qa` / `browse` 中补充真实 DOM / console / network 证据模板

### 完成标志

- 默认主链路具备真实浏览器排查能力
- `browse` 可以从“概念能力”升级为“已验证能力”
- 已提供浏览器专项提取回归脚本，可沉淀真实页面的结构化验证样本

## Direction 3：hooks 从降噪升级为风险治理

目标：

- hooks 不只会降低噪音，还能开始承担基础工程守卫职责

### 重点任务

1. 只读命令自动放行从白名单扩展到风险分级
2. 增加危险命令 `allow / ask / deny` 策略
3. 增加文件写入路径保护
4. 增加 lockfile / secrets / docs 文件保护

### 完成标志

- hooks 能同时覆盖：
  - 状态恢复
  - 高频降噪
  - 基础风险守卫

## Direction 4：文档与 CI 持续同步

目标：

- 让 super-stack 的文档入口、脚本入口和最小自动回归形成稳定闭环

### 重点任务

1. README 只保留正确入口和文档导航
2. 补 `script-architecture` 这类长期维护文档
3. 接入语法检查、unit、integration 级别 CI
4. 让阶段状态、路线图和仓库实现保持同步

### 完成标志

- 新人能快速找到安装、验证、脚本分层入口
- PR 能自动跑一轮最小工程回归
- 文档、ROADMAP、STATE 不再明显脱节

## 4. 当前优先级排序

如果只按“下一步最值”排序，我建议是：

1. 浏览器主链路稳定化
2. hooks 风险分级
3. 文档索引与能力清单持续同步
4. CI 与运行时检查稳态维护
5. 历史生态清理

## 6. 不建议现在做的事

为了避免主线发散，当前不建议优先做这些：

### 6.1 一次性引入过多新技能

原因：

- 技能数量增加很快
- 但如果没有对应验证，很容易形成“看起来很多，其实不可控”

### 6.2 做重型 multi-agent orchestration

原因：

- 当前双宿主基础能力还在稳固阶段
- 更复杂的调度应该等浏览器、hooks、验证体系更稳定后再做

### 6.3 大规模复制参考项目目录

原因：

- 会破坏当前“共享核心 + 薄适配”的主线

## 7. 阶段性里程碑建议

建议后续以这 3 个里程碑判断进展。

### M1：全局工作流稳定

标准：

- 安装 -> 检查 -> 回归链路稳定
- 文档与实现一致
- 核心 hooks 稳定

### M2：前端排障能力可用

标准：

- 至少一侧宿主具备真实 browser evidence 能力
- `browse` 具备专项回归

### M3：工程守卫能力可用

标准：

- hooks 有基础风险分级
- 文件与命令风险有基本治理
- 经验回写开始体系化

## 8. 与参考项目的后续关系

后续继续参考外部项目时，建议坚持这条原则：

- 参考项目提供方向
- super-stack 负责吸收、裁剪、验证、重组

不要再做成：

- 看见一个能力就直接搬

而要做成：

1. 先记录参考点
2. 再判断落在哪一层
3. 补安装与验证
4. 最后进入主线

## 9. 当前结论

当前 `super-stack` 已经从“搭骨架”进入“做工程化增强”的阶段。

最重要的不是继续快速加功能，而是：

- 把高频真实痛点补齐
- 把验证链路做完整
- 把文档沉淀成未来可接管的系统
