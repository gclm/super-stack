# insky-device-sdk Skill Adjustment Signals

## 这份案例要解决什么

这不是一份“项目做了什么”的普通 retrospective 摘要，而是一份面向 super-stack skill 维护的案例记录。

目标是回答：

1. `insky-device-sdk` 在真实使用中反复暴露了哪些 workflow / skill 边界问题
2. 这些问题哪些是项目特有噪音，哪些值得回写到 shared workflow
3. 如果要调 skill，优先应该改哪几个文件

## 为什么选这个案例

`insky-device-sdk` 很适合做 skill 校准案例，因为它同时具备几种容易让 workflow 漂移的条件：

- 仓库很大，但真实任务经常只聚焦到 `insky-cloud`
- 文档、设计、示例代码、迁移中主线代码并存
- 用户常问“当前做到哪里了”“帮我分析状态”“帮我判断数据库是否合理”
- 用户偶尔会直接要求“使用多 agent”

这类项目最容易出现的问题不是“做不出来”，而是：

- 扫描范围过宽
- 阶段判断过重
- 验证口径过于乐观
- 多 agent 升级阈值不稳定

## 观察到的高频模式

### 1. 用户说“项目”，真正想要的通常是“模块”

在这个仓库里，用户表面上会说：

- 分析当前项目
- 看看当前项目做到哪里了
- 分析数据库设计

但落到执行时，真正高频聚焦的是 `insky-cloud`。

典型信号：

- 中途补充“你只需要分析 `insky-cloud` 模块就可以了”
- 最终结论和证据也几乎都锚在 `insky-cloud`

这说明对于大仓库，很多请求并不需要仓库级 `map-codebase` 起手，而是更适合：

- 先识别目标模块
- 再决定是否需要仓库级背景扫描

### 2. `map-codebase` 在已知模块场景里容易扫得过宽

自动任务进度分析和数据库问题分析这两类请求，都存在类似模式：

- 先读 global workflow
- 先读多个 skill
- 先扫仓库结构
- 再扫大量文档和模块
- 最后才真正收口到 `insky-cloud`

这并不意味着结果错误，而是意味着成本偏高、用户等待偏长、噪音偏多。

对于“模块目标明确”的任务，更合适的默认动作应该是：

- 轻量建立仓库背景
- 快速切到目标模块
- 只在被阻塞时回退做更广扫描

### 3. “已实现”和“已验证”之间的差距非常大

这个项目在新路径阶段最有价值的正向信号，是已经开始显式地区分：

- 主链路已实现
- 测试未完全通过
- 真实事件源尚未接入
- 当前只能说明实现深度，不能说明已完成上线闭环

这类表达在复杂项目里非常重要，因为：

- 文档常常领先于实现
- 代码存在未提交工作区改动
- 测试覆盖并不稳定
- 事件接线和运行态接入常常落后于领域逻辑

这说明 `verify` 的方向是对的，但还应该继续把这类分层表达制度化，而不是只依赖回答习惯。

### 4. “用户要求多 agent”不等于应该立刻重度并行

这个项目中确实出现过用户明确要求使用子代理。

记录表明：

- 子代理有时能提供交叉结论
- 但也会带来超时、等待、归并成本
- 很多结论最终仍需要主线 agent 二次核实和收口

所以真正该强化的不是“看见多 agent 就拆”，而是：

- 任务是否能稳定分成只读子问题
- 子结果是否能低成本汇总
- 主线是否会因此更清晰

对“项目状态判断”“数据库问题盘点”“模块完成度分析”这类任务，默认更稳的策略往往是：

- 单 agent 先收口主问题
- 只有在证据缺口明确时再升级

## 对 skill 的直接调整信号

### `map-codebase`

问题：

- 已知目标模块时，仍容易从仓库级入口重扫一遍

建议：

- 增加明确规则：如果用户已经点名模块，先建立模块边界，再决定要不要做仓库级地图
- 输出结构区分：
  - 仓库背景
  - 目标模块结论
  - 与任务直接相关的邻近模块

优先落点：

- `/skills/planning/map-codebase/SKILL.md`

### `verify`

问题：

- 复杂项目里，“代码有了”和“结果被证明了”差距很大，且经常混说

建议：

- 强制鼓励以下口径：
  - 已实现
  - 已验证
  - 未验证
  - 缺口 / 阻塞点
- 对“开发进度判断”“完成度判断”类请求，优先产出验证层面的保留意见，而不是直接给乐观百分比

优先落点：

- `/protocols/verify.md`
- `/skills/quality/verify/SKILL.md`

### 多 agent 升级阈值

问题：

- “用户要求多 agent”容易被误读为“默认值得多 agent”

建议：

- 明确把多 agent 视为升级动作，不是礼貌性默认动作
- 补充负面示例：
  - 单模块状态判断
  - 数据库设计盘点
  - 单链路缺口分析
  - 这些通常先单 agent 收口更稳

优先落点：

- `/AGENTS.md`
- `/protocols/workflow-governance.md`

### `codex-record-retrospective`

问题：

- 路径迁移会影响取证完整性
- 仓库级命中不代表模块级结论可靠

建议：

- 保留 `--project-path-alias` 这类路径别名能力
- retrospective 输出里增加一句：
  - “本项目高频真实目标模块是 X，不等于整个仓库”

优先落点：

- `/skills/planning/codex-record-retrospective/SKILL.md`
- `/skills/planning/codex-record-retrospective/scripts/find_codex_project_records.py`

## 哪些不该过度泛化

以下现象更偏项目特有噪音，不建议直接写成 shared rule：

- `insky-device-sdk` 仓库内同时存在 `device-sdk-examples`、`insky-cloud`、`dji-cloud` 的特殊结构
- 文档与实现并行推进、且设计文档较重的工程风格
- 某些数据库 / 自动任务 / 事件接线问题的具体业务细节

这些可以作为案例背景，但不应直接变成通用规则。

## 推荐优先级

如果基于本案例做 skill 调整，建议优先顺序是：

1. `map-codebase`
2. `verify`
3. 多 agent 升级阈值
4. `codex-record-retrospective` 的模块级提示

## 一句话结论

`insky-device-sdk` 给 super-stack 的关键提醒不是“项目搬过家”，而是：

在大仓库、强模块化、文档和实现并行推进的项目里，skill 最容易失真在三件事上：

- 扫描范围过宽
- 实现与验证混说
- 多 agent 升级过快
